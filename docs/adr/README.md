# Architecture decision records

Use ADRs for durable choices that affect multiple modules, security/privacy policy, release compatibility, or the public product contract.

## Status values

- **Proposed** — under review or awaiting a spike.
- **Accepted** — current implementation direction.
- **Superseded** — replaced by a later ADR; retain for history.
- **Rejected** — considered and not adopted.

## Template

```markdown
# ADR-NNNN: title

- Status: Proposed
- Date: YYYY-MM-DD
- Owners: ...

## Context

## Decision

## Consequences

### Positive

### Negative / tradeoffs

## Alternatives considered

## Validation and rollback

## References
```

Number ADRs sequentially. Update the relevant implementation-plan section when an accepted decision materially changes scope or ordering.
